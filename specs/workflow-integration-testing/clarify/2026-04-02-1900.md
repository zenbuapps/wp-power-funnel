# Clarify Session 2026-04-02 19:00

## Idea

補強工作流引擎的整合測試，涵蓋三個層面：

1. **Action Scheduler 排程鏈路** -- 補完 ActionSchedulerChainingTest 中 6 個 markTestIncomplete 的測試，驗證 NodeDTO::try_execute() 在成功/失敗/延遲節點三種情境下的 AS 排程行為。

2. **Context 跨節點傳遞** -- 驗證 context_callable_set 解析後的 context key 能在節點 params 中被 ParamHelper 以 {{variable}} 格式替換為實際值，確認 EmailNode 的 recipient 從 context 取得後 wp_mail 的收件人正確。

3. **全鏈路 E2E 整合測試** -- 以 ORDER_COMPLETED (pf/trigger/order_completed) 觸發，建立包含 TagUserNode + EmailNode + WaitNode + WebhookNode 的 workflow，驗證整條鏈路走通（含 WaitNode 手動模擬 AS 到期繼續執行）。並測試 YesNoBranchNode 分支路徑（條件成立走 yes、不成立走 no）。

## Q&A

- Q: 需求是否涉及新的 REST API 或資料庫結構變更？ → A: 否，純測試程式碼，無新 API 或 Entity 變更。
- Q: ORDER_COMPLETED 觸發點是否已存在於 ETriggerPoint？ → A: 是，已確認 ETriggerPoint::ORDER_COMPLETED = 'pf/trigger/order_completed'，且 TriggerPointService 已實作觸發邏輯。
- Q: ReplaceHelper null-object bug 在測試中如何繞過？ → A: 現有模式是在 configure_dependencies() 中透過 filter 注入 test_email 節點定義（直接呼叫 wp_mail()），不使用 ParamHelper::replace()。Context 傳遞測試可用此 test_email 節點搭配 wp_mail filter 捕捉實際收件人。
- Q: WebhookNode 在測試中如何 mock 外部 HTTP 請求？ → A: 使用 WordPress 的 pre_http_request filter 攔截 wp_remote_request()，回傳 mock response，不做真實網路請求。
- Q: WaitNode 跨越在測試中如何模擬 Action Scheduler 到期？ → A: 手動呼叫 do_action('power_funnel/workflow/running', ['workflow_id' => $wf_id]) 模擬 AS 排程到期後的觸發行為。
- Q: ActionSchedulerChainingTest 現有 7 個 test methods（非 6 個），是否全部實作？ → A: 是，Issue 說的是 6 個（指 feature 中的 6 個 Example），但測試中實際有 7 個 methods（最後一個 YesNoBranchNode 排程測試為額外新增），全部實作。
