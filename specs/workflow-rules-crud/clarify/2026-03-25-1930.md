# Clarify Session 2026-03-25 19:30

## Tactical 層推導記錄

從已完成的 Strategic 層規格（6 個 .feature + 1 個 .activity）推導 API 規格與 Entity 規格。

## 推導決策

### API 規格（api.yml）

- Q: endpoint 結構 → A: 使用 WordPress REST API `/wp/v2/posts`，非自訂端點。4 個 endpoints 對應 6 個 features。
- Q: edit-workflow-rule 與 save-workflow-rule 是否為同一 endpoint → A: 是，兩者皆透過 POST `/wp/v2/posts/{id}` 更新，差異在於送出的欄位範圍（基本資訊 vs 完整含節點）。
- Q: configure-flow-nodes 是否需要獨立 API → A: 否，節點配置為純前端 React Flow 狀態管理，僅在 save 時一併送出。
- Q: ETriggerPoint 的值格式 → A: 帶有 `pf/trigger/` 前綴（如 `pf/trigger/registration_created`），與 feature 中 `registration_created` 不同，API 層使用完整值。
- Q: WP REST API 更新使用什麼 method → A: WordPress REST API 慣例使用 POST method 更新文章（非 PUT/PATCH）。
- Q: 刪除是否需要 force 參數 → A: 是，`force=true` 跳過回收站直接永久刪除。
- Q: 回應格式 → A: 遵循 WordPress REST API 標準格式（含 title.rendered、meta 物件等），分頁資訊在 response headers（X-WP-Total、X-WP-TotalPages）。

### Entity 規格（erm.dbml）

- Q: NodeDTO 是否為獨立資料表 → A: 否，為 Value Object，以 JSON 陣列形式嵌入 workflow_rules.nodes meta。但為了明確描述結構，在 DBML 中建模為 Table 並標註說明。
- Q: Workflow 執行實例是否納入 → A: 是，workflow_rules 與 workflows 為一對多關聯（一個 Rule 可觸發多個 Workflow 實例），需包含以完整描述資料模型。
- Q: NodeDTO 的屬性是否與 feature 中的 nodeType/label 對應 → A: feature 中的 nodeType/label 為前端 React Flow 展示用概念，後端 NodeDTO 使用 id/node_definition_id/params 結構。API 層需做映射。
