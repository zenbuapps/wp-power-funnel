# Clarify Session 2026-04-01 17:00

## Idea

實作 Power Funnel 工作流引擎中 7 個尚未實作的 NodeDefinition 的 execute() 方法：

1. **LineNode** — 傳送 LINE 文字訊息
2. **SmsNode** — 傳送 SMS
3. **WebhookNode** — 發送 HTTP Webhook
4. **WaitUntilNode** — 等待至特定時間點
5. **TimeWindowNode** — 等待至時間窗口開始
6. **SplitBranchNode** — 多路分支（依 JSON 條件集）
7. **TagUserNode** — 新增/移除用戶標籤

所有節點已在 `register_default_node_definitions` 中註冊、form_fields 已定義，僅 execute() 方法尚未實作（目前丟出 BadMethodCallException）。

## Q&A

- Q: SmsNode filter 回傳值型別？ → A: 選 C。filter 回傳 `array{success: bool, message: string}`，預設值為 `['success' => false, 'message' => 'SMS 發送失敗']`。用戶希望每個 node 都有一樣的回傳值結構，方便管理跟傳遞。
- Q: TimeWindowNode 暫停機制？ → A: 選 A，但更大的架構決策：**所有節點都跑在 Action Scheduler 上**。每個節點成功執行後，引擎統一透過 `as_schedule_single_action(time(), 'power_funnel/workflow/running', ...)` 排程立即執行下一個節點。WaitNode/WaitUntilNode/TimeWindowNode 只是把 timestamp 改成未來時間。影響：(1) NodeDTO::try_execute() 成功路徑需加入排程邏輯，(2) 現有 EmailNode 等不需改 execute() 但引擎層需統一補上，(3) TimeWindowNode/WaitUntilNode 只需回傳正確 timestamp。
- Q: SplitBranchNode condition_value 處理？ → A: 不實作 SplitBranchNode，從本次需求範圍中移除。理由：目前想不到明確的應用場景，YesNoBranchNode 已足夠。SplitBranchNode 留存為枚舉存根（throw BadMethodCallException），待未來有明確場景再實作。
- Q: TagUserNode tags 欄位型別？ → A: 選 B。tags 欄位改為 `type=tags_input`，純字串陣列，user_meta `pf_user_tags` 存 JSON array of string。前端需要一個 tag input UI 元件，不需後端 API。

## 實作決策（由代碼審查推導）

- **WaitUntilNode 過去時間處理**：若 datetime 參數解析後的 timestamp 早於當前時間，視為「立即執行」，使用 `time()` 作為排程時間（與 WaitNode 行為一致）。
- **TimeWindowNode timezone 預設值**：timezone 欄位 `required=false`，未提供時預設使用 `wp_timezone_string()`（WordPress 站台時區設定）。
- **TimeWindowNode 跨日處理**：若 start_time > end_time（例如 22:00 ~ 06:00），視為跨日時間窗口，排程至下一個窗口開始時刻。
- **Action Scheduler 統一排程**：NodeDTO::try_execute() 成功路徑（code 200）需新增 `as_schedule_single_action(time(), 'power_funnel/workflow/running', ...)` 排程邏輯。WaitNode/WaitUntilNode/TimeWindowNode 的 execute() 回傳 200 後，引擎不再二次排程（由節點自身排程未來時間），引擎只負責「非延遲節點」的立即排程。區分方式：WorkflowResultDTO 新增 `scheduled` 布林欄位，節點自行排程後設為 true，引擎據此決定是否需要排程下一步。
