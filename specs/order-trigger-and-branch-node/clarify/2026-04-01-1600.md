# Clarify Session 2026-04-01 16:00

## Idea

這是一個 WordPress 外掛（Power Funnel）的工作流引擎擴充需求，包含兩個功能：

### 功能 1: 新增「訂單完成後」觸發時機點 (Trigger Point)

- 新增 `ETriggerPoint::ORDER_COMPLETED` case，hook value 為 `pf/trigger/order_completed`
- 在 `TriggerPointService` 監聽 WooCommerce `woocommerce_order_status_completed` hook
- 建立 `build_order_context_callable_set()` 和 `resolve_order_context()` 靜態方法
- Context 包含訂單關鍵欄位（`order_id`、`order_total`、`billing_email`、`customer_id`、`line_items` 等）
- 在 `ParamHelper` 擴充 order 相關的 `{{variable}}` 替換支援
- WooCommerce 未啟用時靜默忽略（軟依賴）
- 必須使用 `wc_get_order()` API 相容 HPOS
- 遵循 Serializable Context Callable 模式

### 功能 2: 實裝「是/否分支」節點定義 (YesNoBranchNode)

- 改造工作流引擎核心執行邏輯，支援條件跳轉（非線性執行）
- 實裝 `YesNoBranchNode.execute()`，根據條件判斷結果決定下一個執行節點
- 設計 `form_fields`：讓用戶在 ReactFlow 編輯器中選擇 context key、比較運算子、比較值
- 支援的運算子：`>`、`>=`、`<`、`<=`、`==`、`!=`、`contains`、`not_contains`、`is_empty`、`is_not_empty`
- 「是」和「否」分支各指向不同的後續節點 ID
- 在 `WorkflowResultDTO` 中新增可選的 `next_node_id` 欄位支援跳轉
- 防止節點迴圈（已執行節點不可再次執行）
- 現有線性節點行為完全不變（向下相容）

## Q&A

- Q: resolve_order_context() 回傳的 context 欄位有哪些？ → A: 選 C，完整 9 個欄位：order_id、order_total、billing_email、customer_id、line_items_summary、shipping_address、payment_method、order_date、billing_phone
- Q: ParamHelper 的訂單變數替換機制該怎麼實作？ → A: 選 A，修改 ParamHelper::replace() 新增 $order 物件替換（與 $user、$product 同層級），透過 wc_get_order() 取得 WC_Order 物件。WordPress DB 有物件快取，查詢成本低且能取到完整訂單欄位。
- Q: YesNoBranchNode 的 condition_field 欄位類型？ → A: 選 B，type=select，選項從觸發點的 context keys API 動態帶入。需新增 REST API 端點回傳各觸發點的可用 context keys。新增 STEP:5 與 query-trigger-point-context-keys.feature。
- Q: next_node_id 指向不存在的節點時 Workflow 應如何處理？ → A: 選 A，直接標記 workflow 為 failed，錯誤訊息「找不到目標節點 {next_node_id}」。
- Q: 分支迴圈偵測方式？ → A: 選 A，嚴格禁止重複執行。已在 results 中出現的 node_id 不可再次執行，偵測到即標記 workflow 為 failed，錯誤訊息「偵測到節點迴圈：節點 {node_id} 已執行過」。
