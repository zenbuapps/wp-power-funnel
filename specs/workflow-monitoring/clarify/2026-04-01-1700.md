# Clarify Session 2026-04-01 17:00

## Idea

用戶希望建立「Workflow 執行監控與除錯頁面」。

目前的問題：Workflow 執行資料都有存（pf_workflow CPT + wp_postmeta 存 nodes/results/context），但完全沒有 UI 或 API 可以查看。用戶無法知道 workflow 是否被觸發、節點執行到哪裡、哪裡出錯。

功能目標：

A. Workflow 執行清單頁（List View）
- 列出所有 pf_workflow 實例，支援篩選（status/trigger_point/workflow_rule_id）、搜尋、排序
- 表格呈現：觸發時間、來源規則名稱、觸發點類型、狀態（running/completed/failed）、節點進度（3/5）、耗時

B. Workflow 執行詳情頁（Detail View）
- 以 ReactFlow 唯讀流程圖視覺化每個節點的執行狀態
- 節點狀態圖示：200 成功 / 301 跳過 / 500 失敗 / 進行中 / 未執行
- 點擊節點展開 Drawer：完整 WorkflowResultDTO（code/message/data）、執行時間
- 顯示 workflow context（resolved）和 context_callable_set（原始）

C. 觸發診斷面板（Trigger Diagnostic）
- 視覺化觸發鏈路：業務事件 -> TriggerPointService -> hook 發射 -> WorkflowRule 匹配 -> Context resolve -> Node 執行
- 每個環節顯示成功/失敗狀態
- 解決「workflow 完全沒被觸發時，不知道哪個環節出問題」的痛點

D. 後端 REST API
- GET /power-funnel/v1/workflows -- 清單（分頁+篩選）
- GET /power-funnel/v1/workflows/{id} -- 詳情（含 nodes/results/context）
- GET /power-funnel/v1/workflows/{id}/timeline -- 觸發鏈路時間線

E. 執行時間紀錄增強
- WorkflowResultDTO 新增 executed_at
- Workflow postmeta 新增 started_at、completed_at

建議拆分：
Phase 1: 後端 REST API + WorkflowResultDTO 時間戳增強
Phase 2: 前端 Workflow 清單頁 + 詳情頁（節點狀態視覺化）
Phase 3: 觸發診斷面板（TriggerPointService logging + 前端面板）

## Q&A

- Q: 需求範圍界定 -- 觸發診斷面板（Phase 3）是否納入本次 specs？ → A: 選 C -- 三個 Phase 都做，但詳情頁只放簡化版觸發資訊（用現有的 context_callable_set + trigger_point，不新增 logging 系統）
- Q: API 路徑命名空間 -- 用 WP 標準 REST API、自訂 API、還是混合模式？ → A: 選 B -- 使用自訂 API `/power-funnel/workflows`，遵循現有命名空間慣例，response 封套為 `{code, message, data}`
- Q: 時間戳儲存策略 -- 在 WorkflowResultDTO 加 executed_at、在 Workflow 層級加 started_at/completed_at、還是兩者都加？ → A: 選 B -- 僅在 WorkflowResultDTO 新增 executed_at，Workflow 層級的耗時由 API 從 results 陣列推算
- Q: context_callable_set 在 API 中的回傳格式 → A: 選 B -- 轉換為人類可讀格式，精簡 class path 僅保留 class 短名 + method，保留 params
- Q: 詳情頁 ReactFlow 唯讀流程圖的實作策略 → A: 選 A -- 複用現有 FlowCanvas 元件體系，新增 ReadonlyFlowCanvas wrapper（nodesDraggable=false、nodesConnectable=false），用 CSS overlay 顯示節點執行狀態
- Q: 清單頁搜尋功能搜什麼？ → A: 選 B -- 搜尋節點結果的 message 欄位（模糊搜尋錯誤訊息）。搜尋和篩選並存：搜尋用於關鍵字模糊搜尋 results message，篩選用於精確篩選 status、trigger_point、workflow_rule_id
